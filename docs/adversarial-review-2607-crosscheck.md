## 二、逐条判断

### ✅ 同意且被**低估**：buyer 子串匹配（视角二第 1 条）

实证复刻 `buyer_matcher.py` L117-129 逻辑，跑出比原审查更严重的反例：

```
'Global Fasteners Trading LLC'   -> global_fasteners   (审查断言，成立)
'GF Industrial Supply'           -> global_fasteners   (审查未提)
'Apexon Software Ltd'            -> apex_bolt           (审查未提)
```

后两个是**放大器**：L119-123 把**短别名**（`GF`、`Apex`）和 legal_names 混进同一个双向子串池。
`"GF"` 是 `"GF Industrial Supply"` 的子串 → 毫不相干的公司被匹配走；`"Apex"` 是 `"Apexon"` 的子串 → 跨行业误匹配。
**短别名做子串匹配的误匹配面，比"长公司名互为子串"大一个数量级。**

> 修复方向：(1) 加相似度阈值（长度比 + 编辑距离 / token 重叠）；(2) 把 `aliases` 移出子串池，
> 短别名只允许精确匹配（已有优先级 3 覆盖）；(3) 补负样本测试。

### ✅ 同意但**措辞需修正**：PI/CI 缺强制人工确认（视角一"最严重第 1"）

原审查表述"从'价格填好了'到自动重生成 PI/CI 中间没有强制人工确认关卡"——**在机器校验层面不准确**。

`main.py:508 run_price_update` 实际**有一道自动 precheck 关卡**（L521-524）：回写后跑校验，有 error 就停、
不写 PI/CI/PL，另有 seller 硬校验（L555-561）。

但**实质担忧仍成立**，应改述为：precheck 拦的是"规则可检测的错"（缺重量、格式非法、seller 不存在），
它**结构上拦不住"buyer 抬头匹配错了但字段完整合法"这类语义错误**——恰是子串误匹配制造的错。
且 precheck 是**程序自动过的，不是人看过的**；SKILL.md 无"发送前人工核对抬头/金额"一句。

**修正结论**：不是"没有关卡"，而是"有机器关卡，但没有针对语义错误的人工关卡，且这道机器关卡对最危险的
错误类型（抬头错）恰好是盲的"。

### ✅ 同意并追加一个 bug：canonicalizer 静默直通（视角二第 2 条）

同意静默直通。**追加**：`_normalize_description` L67-70 循环里的 `break`（L70）——一行描述若同时含多个中文品名
（如"六角螺栓配平垫圈"），**只翻译第一个命中的词就 break**，其余中文原样留下。原审查说"字典外的词直通"，
这个 `break` 是"**字典内的词也可能漏译**"，同症状下的第二处失血。

### ⚖️ 同意但**被夸大**：`--use-llm` 隐私风险（视角一"最严重第 3"）

事实成立（grep 确认），但严重性应下调，理由：
1. **默认不开**：`llm_parser.py:33` `use_llm 默认 False`，需显式 `--use-llm` 才外发。是 opt-in-to-risk，非 opt-out。
2. **目标是 Anthropic**——即运行本 Skill 的同一 API 链路，用户数据本就在此链路内，非"泄露给第三方"。
3. 真正该批评的是"**Skill 没告诉用户 `--use-llm` 存在、也没说它和规则模式的差别**"——是文档缺失，
   不该上升到"隐私风险"这种框。

**判断：事实对，定性偏重，降为 P2。**

### ✅ 同意：UUID 回写"实验室级鲁棒，生产级易碎"

UUID 锚点对"行内编辑"确实免疫（重复检测 L78-79、孤儿校验 L95-107 均稳），但整个安全性建立在
"文件不离开 Excel 生态"的假设上。目标用户真实工作流是微信/邮件传 Excel——WPS 另存、手机预览转发、
截图誊抄全都让隐藏列失效。这句判断精准，无可补充。

### ✅ 同意但影响有限：extractor 零测试（视角二"最严重第 3"）

`test_extractor.py` 确实不存在。但**缓解因素**：extractor 产出会经 canonicalizer、buyer_matcher、precheck、
review.json 人工环节多重过滤，非"一错到底"。零测试是技术债，但列进"最严重 3 项"偏高——实际爆炸半径
小于 buyer 误匹配。**详见第四节排序调整。**

### ⚖️ 触发词脆弱性（视角一第 1 条）：成立但最轻，且被漏了一半

裸词 `"init"`/`"setup"`（init/SKILL.md L9）误触发成立，但只是低危（误触发一个交互向导，用户一眼可辨）。
**漏了更重的一半**：见第三节遗漏 2。

## 三、原审查遗漏的、更严重的风险

### 🔴 遗漏 1（代码层，进前三）：`except Exception` 吞掉一切，静默降级污染数据

`llm_parser.py:412` 是 `except Exception as e:`——**不只兜网络错误**。`json.loads`（L399）、schema 缺字段、
`response.content[0]` 越界，全被吞掉后静默 `return _parse_with_rules(doc)`。

后果比"用户不知道降级了"更糟：用户**主动加 `--use-llm`**（通常因这张单结构复杂、规则模式搞不定），
结果 LLM 返回 JSON 稍有格式问题就**悄悄退回它搞不定的规则模式**且不报错——用户以为拿到 LLM 精度，
实际是规则模式降级结果，毫不知情。这让降级从"容灾"变成"数据污染的隐形通道"。

> 修复方向：收窄 except——只对网络/超时/API 错误 fallback；`json.loads`/schema/索引错误应显式报错或
> 至少在返回结果里打降级标记，让上层能提示用户。

### 🔴 遗漏 2（Skill 设计层）：run 的触发词在正常业务对话里高频误撞

原审查只说 `"init"/"setup"`。更严重的是 `run/SKILL.md` 触发词 `"价格填好了"`、`"帮我出单"`、`"做报价"`——
这些是外贸业务员**一天到晚在微信里说的日常话**。误触发 run 会直接启动**对外单证生成链**，
危害等级远高于误触发一个配置向导。

> 修复方向：给 run 触发词加前置条件（须先指定询价单文件 / 须处于项目目录），或改为更明确的显式短语。

### 🔴 遗漏 3（市场层）：见第五节，原审查最大盲区。

### 🟡 遗漏 4（代码层）：`_find_price_column` fallback 到第 6 列是静默错位隐患

`price_updater.py:40-41`，找不到单价列时 **fallback 到硬编码第 6 列（F）**。若用户改了报价单列顺序
（而 UUID 机制的卖点正是"容忍改动"），价格列不在 F 了，代码会**默默从 F 列读数当单价回写**，不报错。
这和审查痛批的"行号 offset 方式"是同一种病，藏在 fallback 里。

> 修复方向：找不到单价列应 `raise PriceUpdateError`，而非 fallback 到固定列。

## 四、"最严重 3 项"统一排序（跨视角）

原审查按视角分别排，未给全局排序。下面按 `爆炸半径 × 静默程度 × 触发概率` 给统一严重性排序
（静默 + 直达客户 = 最危险）：

| 排序 | 问题 | 定位理由 |
|---|---|---|
| **P0-1** | **buyer 子串匹配误配（含短别名放大）** | 静默、字段全合法、precheck 盲区、直达 PI/CI 客户抬头。唯一"会自动发错但没人会发现"的错。**全局第 1。** |
| **P0-2** | **无针对语义错误的人工确认关卡** | P0-1 的"最后一道本可拦住的门"。二者是**因果对**：匹配错 + 没人复核 = 事故。应与 P0-1 绑定修。 |
| **P0-3** | **`except Exception` 静默降级污染数据**（遗漏 1） | 静默、损正确性、专在用户"最需要精度"时触发。**替换掉原审查的"extractor 零测试"。** |
| P1 | canonicalizer 静默直通 + `break` 漏译 | 中英夹杂发客户，尴尬但不致命 |
| P1 | UUID 生态外失效 | 卖点天花板，属"能力边界"非"会出错" |
| P2 | `--use-llm` 隐私零文档 | 默认关、发给同链路 Anthropic，降级 |
| P2 | extractor 零测试 | 有下游多重过滤兜底，爆炸半径被高估 |

**与原审查最大分歧**：把"extractor 零测试"移出前三，换成遗漏的"`except Exception` 静默降级"。
理由——测试缺失是"可能有错"，静默降级是"一定会在特定条件下悄悄出错"，后者期望损失更高。

## 五、市场认可度：原审查最大盲区

原审查三原因（分发渠道错配 / 责任兜底缺失 / 安装门槛）都成立，但共同特征是**全属"外部"原因——
都在讲"用户够不着"**。它没质疑更根本的内部问题：这东西对目标用户是不是真需求、值不值得跨门槛。补两个：

### 🔴 更根本原因 A：自动化的是"低频 + 低痛"环节

第一性原理：1-3 人小外贸团队一天出**几张** PI/CI？通常个位数甚至几天一张。熟练工手填一张 **10-20 分钟**。

工具省的是"每周几次、每次 20 分钟"的活。要求用户付出的是：装 Python 3.12 环境、clone、pip install、
配 Claude Code、学触发词、填 config.yaml、每次核对 review.json……**为省低频任务的 20 分钟，先付几小时
一次性学习成本 + 持续信任成本，ROI 在真实使用频率下不成立。**

外贸业务员真正痛的是**找客户、催款、跟单、货代协调**，不是填单据。填单据烦但不痛。
**这是比"分发"更前置的死因：即使把工具直接送到手上、帮他装好，他也未必天天用。**

> 旁证：本次复核的开发机上连 Python 3.10+ 都没装，项目自身跑不起来（`str | None` 在 3.9 直接报错）。
> 作者本人环境尚且如此。

### 🔴 更根本原因 B：高风险单据场景里，自动化与信任对冲

原审查"责任兜底"讲"作者不担责，竞争不过有 SLA 的 SaaS"。更深一层：**本场景里自动化程度越高，用户越不敢用。**

PI/CI 一字错（抬头、税号、金额、币种）= 真金白银损失或清关纠纷。谨慎的业务员**无论工具多智能都会逐字核对
最终单据**。一旦必须逐字核对，"自动生成"省的时间就被"核对"吃回去——**自动化在高风险单据场景存在天花板：
它不能替用户承担"看一眼"的责任，而"看一眼"恰是最耗时的部分。** 这不是加客服/SLA 能解决的，是任务性质决定的。

这解释了一个悖论：外贸单据这个"看似很适合自动化"的场景，十几年始终没跑出杀手级工具——
**不是没人做，是任务本身的信任结构不允许用户真正把它交出去。**

### 市场判断总评

原审查三原因论证的是"**为什么用户接触不到 / 不敢碰**"，停在"触达"和"信任"层面。
遗漏的 A、B 更硬：**即使触达了、免费送了、帮装好了，工具在目标用户真实频率和风险结构下，
价值也支撑不起使用成本。** 即——原审查在论证"分发失败"，而更前置的结论是"**即使分发成功，
产品-任务契合度本身也不足**"。

## 附录：2026-07-09 第二轮验收与修复记录

## 六、给修复的一句话总结

`--use-llm` 隐私定性偏重）和**一处排序可议**（extractor 零测试不该进前三，应换成遗漏的

市场部分的 A、B 是产品决策而非代码，供作者判断项目定位时参考，不属本轮修复范围。
